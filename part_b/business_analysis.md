Business Case Analysis: Promotion Effectiveness
# B1. Problem Formulation
(a) This problem can be formulated as a supervised machine learning regression task where the objective is to predict the number of items sold at each store for a given month. The target variable is items_sold, which represents sales volume. The input features include store-related attributes such as store size and location type, promotion type, temporal features like month and year, and external factors such as competition density and whether the period falls on a weekend or festival. This is a regression problem because the output is a continuous numerical value, and the goal is to learn the relationship between input features and sales volume to make accurate predictions.

(b) Using items sold as the target variable is more reliable than total revenue because revenue can be influenced by pricing strategies, discounts, and product mix, which may not accurately reflect customer demand. Promotions are typically designed to increase the number of units sold rather than the total monetary value, making sales volume a more direct measure of promotion effectiveness. This illustrates the broader principle that the target variable in a machine learning problem should closely align with the actual business objective being optimized, ensuring that model predictions are meaningful and actionable.

(c) Instead of using a single global model across all stores, a more effective approach would be to use segmented or hierarchical modelling. For example, separate models can be built for urban, semi-urban, and rural stores, or store-specific effects can be incorporated into a unified model. This approach accounts for the fact that customer behavior, competition, and promotion effectiveness vary significantly across locations, leading to more accurate and context-aware predictions.

# B2. Data and EDA Strategy
(a) The data from the four tables—transactions, store attributes, promotion details, and calendar—would be merged using common keys such as store_id and transaction_date. Store attributes would be joined using store_id, while calendar data would be merged using transaction_date to incorporate weekend and festival information. Promotion details would be aligned based on store and time period. The final modelling dataset would have a grain of one row per store per month. Aggregations would include total items sold as the target variable, along with derived metrics such as average basket size, total footfall, and the promotion applied during that month. This structured dataset ensures consistency and relevance for modelling.

(b) Before building a model, several exploratory data analyses would be conducted. First, analyzing the relationship between promotion types and items sold would help identify which promotions are generally more effective. Second, examining sales trends over time would reveal seasonal patterns and potential peaks during festivals or holidays. Third, correlation analysis would help identify relationships between variables and guide feature selection. Fourth, comparing performance across different store types and locations would highlight segmentation opportunities. These analyses would inform feature engineering decisions, such as creating seasonal indicators or interaction features, and help refine the modelling approach.

(c) The imbalance where 80% of transactions occur without promotions could bias the model toward learning patterns associated with non-promotional periods, reducing its ability to accurately capture the impact of promotions. To address this, techniques such as ensuring balanced representation during training, adding a promotion indicator feature, or evaluating model performance separately for promotional and non-promotional periods can be used. This helps the model better understand and predict the effects of promotions.

# B3. Model Evaluation and Deployment
(a) Given that the data is time-ordered, a temporal train-test split should be used, where earlier data is used for training and the most recent data is reserved for testing. A random split is inappropriate because it can lead to data leakage, allowing the model to learn from future information and resulting in overly optimistic performance. Evaluation metrics such as RMSE and MAE should be used. RMSE penalizes larger errors more heavily, making it useful for identifying models that avoid significant prediction mistakes, while MAE provides an easily interpretable measure of average error. Together, these metrics offer a comprehensive view of model performance in a business context.

(b) To understand why the model recommends different promotions for the same store in different months, feature importance analysis can be used. By examining which features most strongly influence predictions, it is possible to identify factors such as seasonality, customer behavior, and external conditions that drive these decisions. For example, in December, higher demand due to festivals may make loyalty-based promotions more effective, while in March, lower demand may require stronger incentives like discounts. Communicating these insights helps the marketing team trust and act on the model’s recommendations.

(c) For deployment, the trained model should be saved using a method such as joblib and reused for monthly predictions without retraining each time. At the start of each month, new data would be collected, preprocessed using the same pipeline, and fed into the model to generate promotion recommendations for each store. Monitoring should be implemented to track prediction accuracy over time, detect changes in data distribution, and identify performance degradation. If significant drops in performance are observed or new patterns emerge, the model should be retrained using updated data to maintain accuracy and relevance.
